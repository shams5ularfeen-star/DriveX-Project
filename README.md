## `01` · Project Overview & Vision

<div align="center">
<img src="https://capsule-render.vercel.app/api?type=soft&color=D4AF37&height=3" width="100%"/>
</div>

**DriveX Pakistan** is a luxury car rental web application that replicates the complete operational footprint of a real-world premium mobility business — across **11 major Pakistani cities** — using nothing but the browser.

The platform delivers an end-to-end customer journey: **discovery → search → inspection → authentication → checkout → confirmation → tracking** — alongside a fully featured admin control panel.

### 🎯 The Problem We Set Out to Solve

> Pakistan's rental-mobility sector is fragmented across informal operators, classified-ad platforms, and traditional offline agencies. Customers seeking premium short-term vehicle access have lacked a unified, transparent digital channel comparable to global benchmarks like **Turo**, **Sixt**, and **Hertz**.

Most rental platforms in Pakistan are:
- Template-driven and visually inconsistent
- Operationally rigid with static, non-transparent pricing
- Lacking any real admin control over fleet, fees, or discounts
- Leaving promo codes exposed and payment flows unfinished

**DriveX was built as the antithesis** — an editorial-grade interface paired with a flexible, dynamic, admin-controlled engine.

### ✨ What Makes DriveX Different

| Feature | Description |
|---------|-------------|
| 🏗️ **Frontend-only, backend-grade** | Zero server cost — every store behaves like a real database with reactive subscriptions and cross-tab synchronization |
| 💡 **Dynamic Pricing Intelligence** | Service fees and weekend/holiday uplifts are layered transparently — users always see the base price, never a hidden surcharge |
| 🔐 **Hidden Promo System** | Codes work silently in the background; the UI never advertises them, preserving luxury-platform exclusivity |
| 🆔 **Self-Healing Order IDs** | Globally unique numeric IDs that survive counter resets, tampering, or device migration |
| 🎨 **Editorial Design** | Black-gold-white palette with Cormorant Garamond + Inter typography, glassmorphism, and gold-shimmer accents |
| 🏙️ **11-City Coverage** | Full operations across Pakistan's major urban and secondary cities |
| 🛡️ **CNIC Verification** | Pakistani national identity verification integrated into the booking pipeline |

## `03` · System Architecture

<div align="center">
<img src="https://capsule-render.vercel.app/api?type=soft&color=D4AF37&height=3" width="100%"/>
</div>

DriveX is structured as **three clean layers**. Each layer has one responsibility, talks to the next through a typed interface, and **never reaches across boundaries**.

```
╔══════════════════════════════════════════════════════════════╗
║                  1. PRESENTATION TIER                        ║
║                                                              ║
║   React server & client components in app/ and components/  ║
║   Pages compose feature components; features compose         ║
║   UI primitives. Next.js App Router handles routing.         ║
╠══════════════════════════════════════════════════════════════╣
║                         ▼  ▼  ▼                              ║
╠══════════════════════════════════════════════════════════════╣
║                  2. DOMAIN / ENGINE TIER                     ║
║                                                              ║
║   Pure TypeScript modules in lib/ — discount engine,         ║
║   fees engine, booking engine, promo engine.                 ║
║   Stateless calculators: inputs in → decisions out.          ║
╠══════════════════════════════════════════════════════════════╣
║                         ▼  ▼  ▼                              ║
╠══════════════════════════════════════════════════════════════╣
║                  3. PERSISTENCE TIER                         ║
║                                                              ║
║   Reactive localStorage stores with snapshot caching,        ║
║   cross-tab storage events, and React subscriptions          ║
║   via useSyncExternalStore.                                  ║
╚══════════════════════════════════════════════════════════════╝
```

### 📁 Folder Organisation

```
drivex-pakistan/
├── app/                        ← Next.js App Router pages
│   ├── (auth)/                 ← Login, register routes
│   ├── (customer)/             ← Browse, book, track
│   ├── (admin)/                ← Fleet, bookings, discounts CRUD
│   ├── globals.css             ← Design tokens (colours, fonts, shadows)
│   └── layout.tsx              ← Root shell
│
├── components/
│   ├── ui/                     ← Design system primitives (shadcn)
│   ├── features/               ← Feature-specific composites
│   └── layouts/                ← Shell, header, footer, navbar
│
├── lib/
│   ├── auth-store.ts           ← Customer authentication
│   ├── admin-auth.ts           ← Operator authentication
│   ├── cars-data.ts            ← Fleet catalog (83+ vehicles)
│   ├── city-store.ts           ← 11-city management
│   ├── bookings-store.ts       ← Booking persistence & order IDs
│   ├── discount-engine.ts      ← Loyalty, bulk, seasonal discounts
│   ├── fees-engine.ts          ← Service fees & uplifts
│   ├── promo-store.ts          ← Hidden promo-code registry
│   └── types/                  ← Shared TypeScript interfaces
│
├── hooks/                      ← Reusable React hooks
└── public/                     ← Static assets
```

### ⚡ Why This Structure Scales

Adding a new feature (say, a **loyalty tier**) requires only three predictable touch points:

1. A new **store** in `lib/`
2. An **engine** modification
3. A **UI component**

No backend deployment. No schema migration. No API versioning — the entire system updates atomically.

## `07` · Business Logic & Pricing Engine

<div align="center">
<img src="https://capsule-render.vercel.app/api?type=soft&color=D4AF37&height=3" width="100%"/>
</div>

Behind the editorial surface sits a **precise, deterministic calculation pipeline**. Every booking flows through the same stages, with admin-controlled levers at every step.

### 💰 Pricing Pipeline (Worked Example)

```
┌─────────────────────────────────────────────────────────┐
│                   PRICING BREAKDOWN                     │
├────────────────────────────────────────┬────────────────┤
│  Base price (PKR 10,000 × 2 days)      │  PKR 20,000    │
│  Loyalty discount (5% returning user)  │ −PKR  1,000    │
│  Service fee (5%, post-discount)       │ +PKR    950    │
│  Weekend uplift (5%, Saturday pickup)  │ +PKR    950    │
├────────────────────────────────────────┼────────────────┤
│  TOTAL                                 │  PKR 20,900    │
└────────────────────────────────────────┴────────────────┘
```

### 🔧 Two-Tier Pricing Architecture

**Tier 1 — Discount Engine (`lib/discount-engine.ts`)**

| Layer | Trigger |
|-------|---------|
| Loyalty Discount | Returning customers (configurable %) |
| Bulk Discount | Multi-car bookings (qty threshold) |
| Category Discount | Targeted by vehicle category |
| Seasonal Discount | Date-range windows (Eid, peak seasons) |
| Promo Code | Single optional code layer |

**Tier 2 — Fees Engine (`lib/fees-engine.ts`)**

| Fee | Default | Override |
|-----|---------|----------|
| Service Fee | 5% | Admin-configurable per city/category |
| Weekend Uplift | 5% | Saturday or Sunday pickups |
| Holiday Uplift | Custom % | Admin-defined named special dates |

### 🆔 Self-Healing Order IDs

Order numbers start at `100001` and increment monotonically. The allocator floors every new ID at:

```
nextId = max(counter + 1, highestExistingOrderNumber + 1, 100001)
```

So even if the counter is **wiped**, **tampered with**, or **restored from a stale backup**, the next ID is guaranteed to be strictly greater than every booking already in storage.

### 🎟️ Hidden Promo Code System

Codes work silently in the background. The UI shows only `"Enter Promo Code"`. All failure paths — unknown code, expired, deactivated, max-usage hit, category mismatch, minimum-order not met — collapse into one single opaque error: **`"Invalid or expired promo code."`**

This prevents enumeration attacks and aligns the experience with high-end booking platforms where codes are distributed through email and partnerships, never openly displayed.

## `08` · Data Management & Persistence

<div align="center">
<img src="https://capsule-render.vercel.app/api?type=soft&color=D4AF37&height=3" width="100%"/>
</div>

`localStorage` is treated as a **JSON document database**. Each store is a typed wrapper that exposes `read()`, `write()`, `subscribe()`, and a `useStore()` React hook.

### 🗄️ Storage Schema

<div align="center">

| localStorage Key | Data Shape | Owner |
|-----------------|-----------|-------|
| `drivex_users_v1` | `User[]` | auth-store |
| `drivex_current_user_v1` | `string \| null` | auth-store |
| `drivex_admins_v1` | `Admin[]` | admin-auth |
| `drivex_current_admin_v1` | `string \| null` | admin-auth |
| `drivex_cars_v1` | `Car[]` (override) | cars-store |
| `drivex_cities_v1` | `City[]` (override) | city-store |
| `drivex_bookings_v1` | `Booking[]` | bookings-store |
| `drivex_order_counter_v1` | `number` | bookings-store |
| `drivex_discount_settings_v1` | `DiscountSettings` | discount-settings |
| `drivex_promos_v1` | `PromoCode[]` | promos-store |
| `drivex_fee_settings_v1` | `FeeSettings` | fees-store |

</div>

> **Versioned keys** (`_v1` suffix) allow safe future schema migrations without wiping existing user data.

### ⚛️ Reactive Store Pattern

```typescript
// Every store exposes this interface:
interface Store<T> {
  read(): T                           // Hydrate typed snapshot from localStorage
  write(next: T): void               // Serialise + notify all subscribers
  subscribe(cb: () => void): void    // Wire into useSyncExternalStore
  useStore(): T                      // React hook — auto-subscribes component
}
```

**Snapshot caching** ensures stable object references — without this, React would see "new" state on every render and trigger an infinite re-render loop. Cross-tab synchronisation is **automatic** via the browser's native `storage` event.

### 🛡️ Integrity Guarantees

- **Append-only bookings** — past records are never overwritten
- **Versioned keys** allow safe future migrations
- **Defensive parsing** — corrupt JSON falls back to typed defaults instead of crashing
- **SSR safety** — every persistence read is guarded with `typeof window === 'undefined'`

## `09` · Engineering Challenges & Solutions

<div align="center">
<img src="https://capsule-render.vercel.app/api?type=soft&color=D4AF37&height=3" width="100%"/>
</div>

Real bugs encountered during development — and the precise fixes that resolved them. Each problem demonstrates the team's **first-principles debugging discipline**.

---

### 🔴 Challenge 1 — Maximum Update Depth Exceeded

<table>
<tr><td><strong>CAUSE</strong></td><td><code>useSyncExternalStore</code> snapshot getters returned a freshly parsed object on every render invocation. Because React's <code>Object.is</code> reference check detected a "new" reference every time, the hook triggered a re-render — which triggered another snapshot call — producing an infinite loop on the checkout page.</td></tr>
<tr><td><strong>FIX</strong></td><td>Module-level snapshot cache keyed by the raw JSON string. The parser runs <em>only</em> when the string actually changes. Writers pre-warm the cache so the next read returns the exact same reference just written. Render count: ∞ → 1.</td></tr>
</table>

---

### 🔴 Challenge 2 — Checkout Payment Regression

<table>
<tr><td><strong>CAUSE</strong></td><td>No <code>try/catch</code> around <code>addBooking()</code>. Storage quota errors in private-mode browsers froze the spinner permanently. Wallet payment methods (JazzCash, EasyPaisa) had no input UI in the initial version, so empty data slipped through validation silently.</td></tr>
<tr><td><strong>FIX</strong></td><td>Four-stage state machine: <code>validating → authorizing → saving → done</code>, each stage labelled on the button. Per-method validation. Inputs disabled during submission. The entire pipeline wrapped in <code>try/catch/finally</code> so any localStorage exception surfaces as a toast and the spinner is always released.</td></tr>
</table>

---

### 🔴 Challenge 3 — Duplicate Order IDs

<table>
<tr><td><strong>CAUSE</strong></td><td>Session-scoped counters allowed different users on different devices to receive identical order numbers — unacceptable for a booking system requiring auditable identifiers.</td></tr>
<tr><td><strong>FIX</strong></td><td>Global counter seeded at <code>100001</code>. Self-healing allocator floors every new ID at <code>max(counter+1, highestExisting+1, 100001)</code>. Guaranteed unique and monotonic even after storage tampering.</td></tr>
</table>

---

### 🔴 Challenge 4 — Image Mapping Integrity Violation

<table>
<tr><td><strong>CAUSE</strong></td><td>A Suzuki Cultus VXR entry used guessed CDN URLs that did not exist in the verified dataset. The original dataset contained only Cultus and Cultus VXL — no VXR variant existed — violating the project's strict 1-to-1 image-to-car policy.</td></tr>
<tr><td><strong>FIX</strong></td><td>Strict per-car image policy enforced. The unverified VXR entry was removed entirely. The misassigned image was repurposed as a homepage decorative banner — never as a listing image. Fleet count, listings, and admin tables all reconcile automatically.</td></tr>
</table>

---

### 🔴 Challenge 5 — Promo Code Over-Exposure

<table>
<tr><td><strong>CAUSE</strong></td><td>Suggested promo codes (e.g. <em>"Try DRIVEX10 · WELCOME20 · WEEKEND15"</em>) were displayed directly beneath the input field, trivialising the promotion mechanism and enabling trivial enumeration attacks.</td></tr>
<tr><td><strong>FIX</strong></td><td>Hidden by design. The input shows only <em>"Enter Promo Code"</em>. All failure paths collapse into a single neutral error: <em>"Invalid or expired promo code."</em> — preventing enumeration while preserving exclusivity.</td></tr>
</table>

---

### 🔴 Challenge 6 — UI/UX Inconsistencies

<table>
<tr><td><strong>CAUSE</strong></td><td>Booking summary occasionally merged service fee into the daily rate. Success screen missing price breakdown. Payment-method labels absent for newly-introduced rails. Buttons remained active during in-flight submission simulations.</td></tr>
<tr><td><strong>FIX</strong></td><td>Full integrity sweep: itemised line-item breakdown, extended <code>paymentLabel</code> switch covering all rails, disabled states during in-flight requests, animated success ring + gold-shimmer order number + confetti on the confirmation screen.</td></tr>
</table>

---

### 📊 Problems & Solutions Summary

| # | Problem | Root Cause | Resolution |
|---|---------|-----------|-----------|
| 1 | React infinite loop | `getSnapshot` returned new object on every call | Module-level snapshot cache keyed on raw JSON string |
| 2 | Checkout payment freeze | Missing `try/catch`; no wallet input UI | 4-stage state machine + per-method validation |
| 3 | Duplicate order IDs | Session-scoped counters per user | Global counter with self-healing `max()` floor |
| 4 | Wrong car images | Guessed CDN URLs for non-existent variant | Entry removed; image repurposed as decorative banner |
| 5 | Promo code enumeration | Example codes visible in UI | All suggestions removed; opaque error messages |
| 6 | UI inconsistencies | Service fee merged, missing labels, active buttons | Full integrity sweep across checkout and success flows |

## `10` · System Evaluation

<div align="center">
<img src="https://capsule-render.vercel.app/api?type=soft&color=D4AF37&height=3" width="100%"/>
</div>

### ⚡ Performance

The application achieves a **sub-second time-to-interactive** on modern Chromium. Key optimisations:
- Tailwind's content-aware purge keeps bundle size minimal
- Next.js per-route code splitting
- Snapshot-cache pattern eliminates avoidable re-renders
- CSS-only animations (confetti, gold glow, fade-in) avoid the JS animation tax

### ✅ Strengths

| Strength | Detail |
|----------|--------|
| **End-to-End Fidelity** | Every pathway — registration through booking to admin approval — completes with zero external dependencies |
| **Architectural Cleanliness** | The framework-agnostic domain layer makes the system trivially portable to a real backend |
| **Operator Parity** | The admin panel exposes controls a real operations team would expect |
| **Defensive Engineering** | Order IDs, snapshot caches, and `try/catch` boundaries handle failures that normally require server intervention |
| **Brand Coherence** | Gold-on-black visual system applied uniformly across customer and admin surfaces |

### ⚠️ Limitations

| Limitation | Impact |
|-----------|--------|
| **Single-Origin Persistence** | `localStorage` is per-browser; bookings can't be shared across devices |
| **Simulated Payments** | No money moves; authorizing stage is a timed `await` |
| **5 MB Storage Quota** | Large booking histories could approach browser's localStorage ceiling |
| **No Server-Side Validation** | Determined users could edit storage by hand |
| **No Audit Log** | Writes are destructive; no append-only audit semantics |
